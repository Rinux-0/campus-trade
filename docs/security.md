# 安全与并发说明

## 八、安全性

### 1. 如何防止普通用户删除数据

本系统为单用户本地项目，不设用户登录系统。若需限制删除权限，可采取以下措施：

- **角色控制**：区分管理员与普通用户角色，删除操作需验证管理员身份（如 JWT token 携带 role 字段）
- **后端中间件**：在 Axum 的 DELETE 路由上添加中间件层，校验请求头中的 token 是否具有管理员权限，无权限则返回 403 Forbidden
- **前端隐藏**：普通用户登录后前端不渲染删除按钮，但仅前端隐藏不够，必须以服务端鉴权为准
- **数据库层**：可创建 PostgreSQL 只读角色，应用程序以受限用户连接，仅管理接口使用高权限连接

### 2. 如何限制用户只能查询数据

- **最小权限原则**：为普通查询用户创建 PostgreSQL 只读角色（`GRANT SELECT ON ALL TABLES`），仅授予 SELECT 权限，禁止 INSERT/UPDATE/DELETE
- **API 层隔离**：后端分为只读接口和写入接口，只读接口使用受限数据库连接池，写入接口使用完整权限连接
- **路由级鉴权**：查询类 GET 路由不做权限校验（或仅校验登录态），修改类 POST/PUT/DELETE 路由校验管理员角色

## 九、并发与恢复

### 1. 两个用户同时购买同一商品会出现什么问题

若不加锁，典型竞态条件如下：

```
时间线  用户A              用户B
T1     SELECT status=0     SELECT status=0  (都读到未售)
T2     INSERT order        INSERT order    (都插入了订单)
T3     UPDATE status=1     UPDATE status=1
```

后果：同一商品被卖出两次，`orders` 表中同一 `item_id` 出现两条记录，违反"每商品最多交易一次"的约束。若 `orders.item_id` 设有 UNIQUE 约束，后提交的事务会因唯一约束冲突而失败，但用户体验差。

### 2. 如何解决

本系统已实现的方案：

- **数据库事务**：购买操作包裹在 `BEGIN...COMMIT` 事务中，保证原子性
- **行级锁 (SELECT FOR UPDATE)**：事务开始时对 `item` 行加排他锁，阻止其他事务并发修改同一行
- **状态检查**：取锁后检查 `status=0`，若已为 1 则返回 409 Conflict
- **顺序执行**：两个并发请求由于行锁互斥，后一个必须等待前一个提交后才能执行，此时读到 `status=1`，返回错误

### 3. 如果系统崩溃，如何恢复订单数据

- **事务原子性**：PostgreSQL 保证事务的 ACID 特性。若崩溃发生在事务提交前，所有修改自动回滚，数据保持一致性
- **WAL 日志**：PostgreSQL 的预写日志 (Write-Ahead Log) 记录了所有变更。崩溃重启后，已提交但未写入数据页的事务通过 WAL 重做 (REDO) 恢复，未提交的事务通过 WAL 回滚 (UNDO) 丢弃
- **备份策略**：定期 `pg_dump` 导出 SQL 文本备份，或启用 PostgreSQL 的持续归档 (Continuous Archiving) + 时间点恢复 (PITR)
- **业务层幂等**：订单 ID 基于 MAX(order_id)+1 生成，若担心重复，可将 `order_id` 的生成改为 UUID 或使用数据库序列，确保崩溃重试不产生冲突
